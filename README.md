# TeamSpace

## Краткое описание проекта

**TeamSpace** — настольное приложение, разработанное в рамках лабораторной работы № 3 по дисциплине «Технологии программирования».

Проект объединяет аналитические модули трёх участников команды в едином пользовательском интерфейсе. На главном экране отображаются карточки вариантов, с помощью которых можно перейти к соответствующему модулю анализа статистических данных.

В приложении реализованы:

- единое главное меню выбора аналитического модуля;
- загрузка исходных данных из CSV-файлов;
- отображение статистических данных в табличном виде;
- расчёт аналитических показателей;
- построение графиков;
- прогнозирование методом скользящей средней;
- экспорт графических результатов в формат PNG;
- запуск отдельного подключаемого приложения одного из участников.

## Общая организация приложения

Главное меню приложения реализовано в проекте `TeamSpace`. Оно содержит три карточки участников, каждая из которых предоставляет доступ к соответствующему аналитическому модулю.

Способ открытия модулей отличается в зависимости от их реализации:

| Модуль | Способ запуска |
|--------|---------------|
| Анализ доли плохих дорог | Открывается как форма `BadRoadsForm` внутри приложения `TeamSpace` |
| Анализ процента детей, рождённых вне брака | Запускается как отдельное приложение `BirthsOutOfWedlockApp.exe` |
| Анализ потребительских расходов | Открывается как форма `ConsumerExpensesForm` внутри приложения `TeamSpace` |

Для внешнего модуля предусмотрен автоматический поиск собранного исполняемого файла в каталогах `Debug` и `Release`.

## Описание модулей

### Вариант 16 — Анализ доли плохих дорог по субъектам РФ

Модуль предназначен для анализа статистических данных о доле автомобильных дорог, не соответствующих нормативным требованиям, в различных субъектах Российской Федерации.

Основные возможности:

- загрузка данных из CSV-файла;
- отображение исходных данных в таблице;
- выбор региона для анализа;
- получение списка доступных регионов;
- построение графика изменения показателя по годам;
- определение региона с максимальным снижением доли плохих дорог;
- определение региона с минимальным положительным снижением доли плохих дорог;
- применение прогнозирования методом скользящей средней.

Для хранения данных используется модель `RoadData`, содержащая:

- год наблюдения;
- название региона;
- процент плохих дорог.

Положительное снижение процента плохих дорог рассматривается как улучшение состояния дорожной инфраструктуры.

---

### Вариант 17 — Анализ процента детей, рождённых вне брака

Модуль предназначен для обработки и анализа статистических данных о проценте детей, рождённых вне зарегистрированного брака.

Основные возможности модуля:

- загрузка статистических данных из CSV-файла;
- отображение исходных значений в таблице;
- построение графика изменения показателя по годам;
- анализ динамики показателя;
- построение прогноза на последующие периоды.

Данный модуль реализован как отдельный проект `BirthsOutOfWedlockApp` и запускается из главного меню приложения `TeamSpace`.

Для корректного запуска модуля необходимо предварительно собрать проект `BirthsOutOfWedlockApp`, чтобы был создан файл:

```text
BirthsOutOfWedlockApp\BirthsOutOfWedlockApp\bin\Debug\net8.0-windows\BirthsOutOfWedlockApp.exe
```

или файл конфигурации `Release`:

```text
BirthsOutOfWedlockApp\BirthsOutOfWedlockApp\bin\Release\net8.0-windows\BirthsOutOfWedlockApp.exe
```

---

### Вариант 18 — Анализ потребительских расходов

Модуль предназначен для загрузки, анализа и прогнозирования данных о потребительских расходах по годам.

Основные возможности:

- загрузка CSV-файла с данными о расходах;
- отображение исходных значений в таблице;
- выбор анализируемого показателя;
- расчёт максимального процентного изменения;
- расчёт минимального процентного изменения;
- отображение количества доступных периодов;
- выбор периода скользящей средней;
- выбор количества будущих лет прогнозирования;
- построение общего графика фактических и прогнозных значений;
- выделение прогнозной области отдельным цветом;
- экспорт построенного графика в формат PNG.

Поддерживаемые показатели:

- общие расходы;
- продукты питания;
- одежда и обувь;
- жилищные услуги;
- транспорт;
- здравоохранение;
- образование.

Для хранения одной строки данных используется модель `ConsumerExpenseData`. Для передачи результатов расчёта статистики используется модель `ConsumerExpenseStatistics`.

## Прогнозирование методом скользящей средней

Для прогнозирования временных рядов в приложении используется метод экстраполяции по скользящей средней.

Прогнозное значение следующего периода вычисляется как среднее арифметическое нескольких последних значений ряда:

```text
Прогноз = (значение1 + значение2 + ... + значениеN) / N
```

где `N` — выбранный пользователем период скользящей средней.

При построении прогноза на несколько периодов каждое новое рассчитанное значение добавляется в рабочий ряд и используется при вычислении последующих прогнозных значений.

В проекте сервис прогнозирования поддерживает два числовых типа:

- `double` — для статистических процентных показателей;
- `decimal` — для денежных данных потребительских расходов.

Использование `decimal` для расходов позволяет сохранять точность финансовых вычислений.

## Работа с CSV-файлами

Для загрузки исходных данных используется сервис `FileParser`.

Сервис выполняет:

- проверку существования файла;
- чтение строк в кодировке UTF-8;
- удаление пустых строк;
- автоматическое определение разделителя CSV-файла;
- обработку значений, заключённых в кавычки;
- преобразование текстовых значений в числовые типы;
- проверку обязательных столбцов;
- формирование объектов моделей;
- сортировку загруженных данных по году.

Поддерживаемые разделители CSV-файла:

- точка с запятой `;`;
- запятая `,`.

### Формат CSV-файла для модуля потребительских расходов

Файл должен содержать следующие столбцы:

```text
Год;Общие расходы;Продукты питания;Одежда и обувь;Жилищные услуги;Транспорт;Здравоохранение;Образование
```

Пример строки данных:

```text
2010;250000;85000;19000;42000;27000;16000;12000
```

Файл модуля потребительских расходов должен располагаться в каталоге:

```text
TeamSpace\Data\consumer_expenses.csv
```

## Архитектура проекта

Проект организован с использованием разделения на модели, сервисы и формы.

### Models

Классы моделей предназначены для хранения данных и результатов расчётов.

| Класс | Назначение |
|------|------------|
| `RoadData` | Хранение данных о состоянии дорог по региону и году |
| `ConsumerExpenseData` | Хранение значений потребительских расходов за один год |
| `ConsumerExpenseStatistics` | Хранение рассчитанной статистики по расходам |

### Services

Сервисные классы содержат логику обработки данных и вычислений.

| Класс | Назначение |
|------|------------|
| `FileParser` | Чтение и преобразование CSV-файлов в объекты моделей |
| `StatisticsCalculator` | Расчёт статистических показателей для аналитических модулей |
| `MovingAverageForecast` | Расчёт скользящей средней и построение прогноза |

### View

Формы и визуальные компоненты отвечают за взаимодействие с пользователем и отображение результатов.

| Класс | Назначение |
|------|------------|
| `MainMenuForm` | Главное меню выбора аналитического модуля |
| `BadRoadsForm` | Форма анализа доли плохих дорог |
| `ConsumerExpensesForm` | Форма анализа потребительских расходов |
| `ExpensesChartControl` | Пользовательский график фактических и прогнозных значений |
| `SmoothMenuButton` | Пользовательская стилизованная кнопка |

Такое разделение позволяет отделить пользовательский интерфейс от вычислительной логики, упростить сопровождение программы и повторно использовать общие методы обработки данных.

## Инструкция по запуску

### 1. Клонирование репозитория

Выполнить команду:

```bash
git clone https://github.com/KateSkoby/TeamSpace.git
```

После клонирования перейти в каталог репозитория:

```bash
cd TeamSpace
```

### 2. Открытие основного решения

Открыть в Visual Studio файл:

```text
TeamSpace.slnx
```

Если в используемой версии проекта присутствует классический файл решения, необходимо открыть:

```text
TeamSpace.sln
```

### 3. Сборка внешнего модуля Лёши

Для запуска модуля анализа процента детей, рождённых вне брака необходимо предварительно собрать отдельный проект.

Открыть решение:

```text
BirthsOutOfWedlockApp\BirthsOutOfWedlockApp.sln
```

В Visual Studio выполнить:

```text
Сборка → Собрать решение
```

После сборки должен появиться файл:

```text
BirthsOutOfWedlockApp\BirthsOutOfWedlockApp\bin\Debug\net8.0-windows\BirthsOutOfWedlockApp.exe
```

### 4. Сборка и запуск основного приложения

Вернуться к решению `TeamSpace` и выполнить:

```text
Сборка → Собрать решение
```

После успешной сборки запустить проект `TeamSpace`.

### 5. Выбор модуля

В главном меню выбрать требуемую карточку:

- **Вика — вариант 16** — анализ доли плохих дорог;
- **Лёша — вариант 17** — анализ процента детей, рождённых вне брака;
- **Катя — вариант 18** — анализ потребительских расходов.

## Использование модуля потребительских расходов

1. Открыть карточку **«Катя — вариант 18»**.
2. Нажать кнопку **«Открыть CSV»**.
3. После загрузки данных выбрать анализируемый показатель.
4. Выбрать период скользящей средней `n`.
5. Указать количество будущих лет прогноза.
6. Нажать кнопку **«Построить график»**.
7. При необходимости сохранить результат кнопкой **«Экспорт в PNG»**.

На графике отображаются:

- фактические данные;
- прогнозные значения;
- отдельная визуально выделенная область прогноза;
- легенда;
- подписи годов и шкала значений.

## Обработка ошибок

В приложении предусмотрена обработка основных ошибочных ситуаций:

- отсутствие CSV-файла;
- отсутствие обязательных столбцов в файле;
- некорректные числовые значения;
- недостаточное количество данных для прогнозирования;
- попытка экспорта графика до его построения;
- отсутствие собранного файла внешнего приложения;
- ошибки запуска отдельного модуля.

При возникновении ошибки пользователю отображается сообщение с описанием причины и возможным способом её устранения.

## Организация работы с Pull Requests

Для объединения результатов командной разработки применяется следующий порядок работы:

1. Каждый участник разрабатывает свой модуль в собственной `feature`-ветке.
2. После завершения функциональности создаётся Pull Request из `feature`-ветки в ветку `dev`.
3. После проверки и разрешения возможных конфликтов изменения объединяются в `dev`.
4. После интеграции и тестирования всех модулей создаётся Pull Request из `dev` в `main`.
5. Ветка `main` используется как итоговая стабильная версия проекта.

Такая схема позволяет выполнять параллельную разработку, сохранять историю изменений и проверять совместимость модулей перед формированием итоговой версии.
